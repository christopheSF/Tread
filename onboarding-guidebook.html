1
Day 0–3 · 45–60 min

Kickoff & Success Definition

The customer just signed. This call turns a contract into a plan: what success means in their words, how their account should be structured, what data we need, and when go-live happens.

CSM (lead)Implementation leadCustomer: owner / ops managerCustomer: dispatcherCustomer: biller (optional)

Before the call — CS first steps

  • Create the customer accountProvision the company in Tread as soon as the deal is Closed Won.
  • Create user accountsUse the pattern app_support+[customer name]@tread.io for the internal support user.
  • Send the Day-0 Welcome EmailVersion A (new account) or Version B (already on Tread) — see the email chain. Log the closed date, set Status = Onboarding in the tracker.
  • Review the onboarding questionnaireIf sales filled the onboarding intake, read it — the success goal and billing answers shape this whole call.

Agenda — run it in this order

  • Define success in the customer's words"When we talk again in 30 days, what will make you say this worked?" Write the answer down verbatim — you'll replay it on Call 3.
  • Map the account structureOne company, or parent + child accounts? Subsidiaries that dispatch independently (like Igel's DBM trucks) belong in a child account so roles stay clean later.
  • Agree the minimum data setTo create the first dispatch: 5 frequent pickup sites, 3 customers, 3 materials — plus equipment types and users. Everything else can come later.
  • How they'll dispatchFrom Projects each time, copy & edit yesterday's order, or duplicate templates? Pick what matches how their dispatcher already thinks.
  • Which numbers they run onRunning orders vs load count vs tonnage; which references billing needs later (dispatch #, order PO, zones, phase codes). Set defaults now — e.g. Running as the project default if trucks run all day.
  • Set the go-live target & book Call 2Aggressive is fine (Igel targeted ~2 weeks) — but book the training call before you hang up.
✅ What good looks like (real example — George Igel kickoff) By end of call: DBM subsidiary set up as a child account, foreman visibility scoped to their own orders, Running set as the project default, a go-live site chosen, and an on-site training date on the calendar within two weeks.
📋 Capture as you go Log everything in the Onboarding/CS Checklist (tracker): Closed Date, Segment (Enterprise/MM), CS Owner, and flip milestone 1 as data lands.

2
Day 7–14 · 60–90 min (or on-site)

Training: Dispatch, Field Capture & Vendors

The core training. One live walkthrough of the whole loop — dispatch goes out, work happens, tickets and hours come back, the foreman signs off. Then vendors and drivers get invited so real work can start.

CSM / Implementation (lead)Dispatcher(s)Lead foreman (train-the-trainer)Vendors & drivers (invite phase)

Agenda — the end-to-end loop

  • Create a Project → Order → dispatch it, liveUse their real data from Call 1, not demo data. Show their preferred dispatch style (from-project / copy-edit / duplicate) and default sites.
  • Driver mobile app walkthroughAccepting a job, logging loads, uploading the scale ticket photo. Share the Driver App one-pager.
  • Field capture: tickets & hoursWhat gets captured per load — material tickets (OCR reads date/quantity), job hours, shift hours, add-ons. Decide whether drivers may edit hours before submitting.
  • Material linkingConfirm materials on orders link to the master data list so tickets reconcile automatically.
  • Foreman approval gateSet foremen at the order level (not project default) so each sees only their own orders. Phase codes entered at approval — if they're mandatory for the customer, set up an exception report to flag approvals missing one.
  • Invite vendors & driversAdd subhaulers under Settings → Vendors, then run the Day-10 invitation flow: customer forwards our templated email/SMS to every vendor. Confirm acceptance within 3 days.
  • Dispatch a real job before you leaveNot a test. The single strongest predictor of a healthy onboarding is a live dispatch during or right after this call.
⚠️ This is where "trained, not yet dispatching" happens. Many customers get trained and then don't send a real dispatch for weeks. If the call ends without live work scheduled in Tread, book the 15-minute "first dispatch" follow-up before hanging up.
✅ What good looks like (real example — Igel go-live) On-site session structured in three blocks: dispatchers first, then foremen (train-the-trainer with the lead foreman), then vendors/drivers — all on one chosen go-live site, with jobs flowing the same day.

Day 21–30 · 60 min

Billing, Reporting & the 30-Day Review

The payoff call. Approvals now feed billing: payables, receivables, settlements, invoices and exports. Close the first billing cycle cleanly, run the first reports, then review the success goal from Call 1 — in the customer's own words.

CSM (lead)Billing / accounting leadOwner / ops managerDataVinci (if custom reports)

Agenda — close the loop on money

  • Walk the first billing cycle checklistTicket review configured · driver & vendor payables set up · settlement add-ons in place · one test export run (CSV/PDF) · foreman approvals enabled. All five before month-end.
  • Generate pay statements & invoices from approved workShow that only approved loads flow to billing — the approval gate from Call 2 is what makes the invoice trustworthy.
  • Confirm the billing system of recordIs Tread the source of truth, or does an export feed their accounting system? If they need a specific file format (Excel/CSV columns), capture it and open a REP ticket for DataVinci.
  • Set up rates if pricing variesProject rates match customer/vendor/driver rates automatically by site, material, and equipment. Show the rate history log for auditability.
  • Run the first real reportsP&L summary, driver payroll, job cost. Establish a monthly baseline now.
  • Integrations check (Enterprise)Vista export, ERP sync, accounting flows, telematics. Flag open gaps with owners so nothing is lost after the 30-day mark.
  • The 30-day reviewReplay the success definition from Call 1. What's working, what's rough, and the next 30–60 days: advanced dispatching, bulk actions, project rates, integrations.
✅ What good looks like (real example — Thunderbolt billing session) Report reviewed line by line with the biller on the call: unique Load ID per row, approved-loads-only filter, and an approved-date filter so exports can't double-bill. The customer left with a plan to fully migrate off their legacy system and stop dual-logging.
⚠️ Watch for billing data quality traps. Loads with null/zero quantities (never approved, or incomplete tickets), backdated tickets approved after an export already ran, and unit-of-measure mixes (tons vs yards vs hours vs loads). The approved-date filter and a review of stragglers before go-live prevent the worst of it.

Tell success stories — and share the future

End Call 3 on momentum. Two stories that land with almost every customer:

Phase-code compliance

A contractor was losing hundreds of thousands of dollars a year to unallocated costs from missing phase codes on paper tickets. With foreman approvals requiring phase codes plus an exception report, unallocated cost drops to what the report catches — daily, not at month-end.

Kill the dual-logging

Customers migrating from a legacy dispatch system often run both in parallel "to be safe" — doubling driver workload and creating mismatches. A clean approved-loads export with a date filter is what finally lets them switch off the old system.

The future: once the 10 milestones are done, the next 90 days open up — advanced dispatching (running loads, bulk actions, copy/edit), project rates at scale, custom reporting through DataVinci, and deeper ERP/telematics integrations.
