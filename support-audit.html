Section 02

Category Deep Dives

Each category below details the exact issues that come in, the actions support takes, and the standard response patterns that produce the fastest, most consistent resolutions.

Category 01
Account Support & Access
17.6%of support

Customer portal invites, contact info updates, permission changes, settings that don't save, and order/project configuration that requires backend intervention.

Common issues
Expired portal invite link Add/change admin role Pickup location reverts Jobs linking to each other Change customer on order Equipment type missing Contact info update
Actions triggered
  • Resend invite email + set new admin role in backend
  • Update phone/email directly in account settings
  • Explain system limitation + offer workaround
  • File Linear bug when behavior is unexpected
Standard response

"Hi [Name], will check on this."

"We've resent the invite to [email] and added [name] as admin. If you have a phone number for them, share it and we'll update their profile too."

"At this time, [action] isn't possible once [condition]. We can make the change after the job completes — let us know if that works."

Category 02
Driver Management
16.3%of support

The single most technically actionable category. Most tickets require backend access to detach a phone number from a previous company or move a driver between accounts.

Sub-type breakdown
Phone number already taken (prior company)~60%
Phone number rejected as invalid~15%
Driver added to wrong account type~15%
Phone number update request~10%
Actions triggered
  • Detach phone number from previous account in backend
  • Move driver between company / vendor profiles
  • For invalid numbers: verify with carrier, file Linear bug CUS-114 ✓ resolved
  • Phone updates: bot collects NativeForms, agent executes
Standard response — phone taken

"Hi [Name], looking at the driver details now. Which account should we add them to?"

"Done — we've detached the driver from their previous account. Please try adding them again."

Standard response — phone update (bot)

"Please complete this form so we can find the driver and update their number." [NativeForms link]

"Once the form is submitted we'll follow up, or let you know it's been updated."

Category 03
Product Support (How-to)
15.4%of support

Questions about how the platform works — where to find features, what's possible, and requests for data that require backend or SMS-provider access.

Common issues
Can't find reports section Export messaging history Inactive materials still show Duplicate job vs order Ticket Review deprecated
Actions triggered
  • Navigate customer to correct section with screenshot
  • Extract SMS logs from Twilio/provider, send as CSV
  • Explain limitation + provide workaround
  • Log to product team when deprecated feature causes confusion
Standard response

"Please reload the page — you'll see [feature] in the [section] menu." [screenshot]

"We don't store that in-app, but let me pull it from our provider. Can you share the phone numbers and date range?"

"[Feature] isn't available today. The closest option is [workaround]. I've flagged it for the product team."

Category 04
Login / Get Verification Code
8.6%of support

Drivers or dispatchers who can't log in because they're not receiving the SMS verification code, or whose accounts don't yet exist in the system.

Common issues
Not receiving SMS code Account not found Can't set up account
Actions triggered
  • Bot routes to NativeForms form to collect phone number
  • Agent looks up whether account exists in backend
  • If no account: coordinates with dispatcher to create one
  • Resend verification SMS manually if needed
Standard response

"Please complete this form so we can verify whether your account exists or create a new one. If you've already completed it, share the phone number you used."

"Your account is set up under [number]. The code should arrive within seconds — please check your signal. If it still doesn't come through, we'll resend it manually."

"I don't see an account linked to that number. Your dispatcher may need to add you first — can you share their name or company?"

Category 10

1.8%of support

Section 03

Inbox Noise

📧

Unsolicited Email — 56 conversations (18.7%)

Outbound sales emails from third-party vendors (Aloware, etc.) arriving via the email channel. They consume inbox space and agent attention with zero resolution value.

→ Recommendation: Auto-close rule for email-channel conversations where the sender is not a known Tread contact, or filter at the inbox level before routing.

🔇

Non-customer / No Response — 23 conversations (7.7%)

Drivers looking for work ("Looking for Work" tag), vague greetings with no follow-up, and out-of-hours contacts who never replied when the team responded.

→ Recommendation: Auto-close after 48h of no customer reply + a standard "are you still there?" check-in at 24h. Tag "Looking for Work" contacts with a canned redirect.

Section 04

Key Observations & Recommendations

Patterns that emerge across all 300 conversations — structural opportunities to reduce volume, improve resolution speed, and close the loop with customers more consistently.

🔑

"Phone number already taken" is the #1 solvable ticket

It appears in ~60% of Driver Management tickets and always requires backend access. A clearer in-app error message — "This driver is linked to another account. Contact support to transfer them." — plus a one-click transfer flow would materially reduce this category's volume.

🤖

Bot triage is effective but narrow

The TREAD bot handles two intents well (driver phone update, account verification) via NativeForms. Expanding structured intake to cover "add driver — phone taken", "can't log in", and "portal access" could deflect ~30% of current agent load before the first human reply.

📊

Reporting questions spike after feature releases

The mileage report ticket is a pattern: users discover new features via support chat after a UI update. In-app tooltips or a "What's New" banner on the Reports page would convert these into zero-touch resolutions.

🔗

Linear integration is working — closing the loop is not

Bugs are captured and linked (PRO-1589, TRE-13568, CUS-114) and at least one was resolved and communicated back (CUS-114). The gap is consistency: some customers are left waiting after a bug is filed with no follow-up when the fix ships.

⚠️

Stalled open conversations need attention

The Approvals/Timekeeping ticket (TRE-13568 — shift date edit) is still open and unresolved, with a customer who expressed payroll urgency. This pattern — Linear bug filed, customer left waiting — is the most common friction point in the current queue.

🌐

Spanish-speaking drivers are an underserved segment

App Storage and NAR conversations show Spanish-speaking drivers reaching support outside business hours. Cherry's Spanish responses (5★ rating earned) are effective. A Spanish-language bot path and help article set would extend this coverage asynchronously.

📦

56 spam emails are a pure inbox tax

18.7% of all conversations are unsolicited vendor emails. An allowlist-based auto-close rule (only known contacts route to inbox) would eliminate this noise immediately without any product change.

⚡

Response times are a genuine strength

Agents (Marco, Cherry, Ann) consistently respond within 1–2 minutes of assignment. The bot assigns within seconds. This speed is a real differentiator — the opportunity is to match that speed with more consistent resolution depth, not just acknowledgment.
