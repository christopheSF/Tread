⚠ Architecture Gap Discovered — Accounts for 51 Blocked Tests

The spec was written assuming orders have order-scoped SRS records in addition to project-scoped ones. During testing, GET /orders/{id}/site_rate_schedules returned net::ERR_FAILED for every order tested. The endpoint does not exist (or is unimplemented) in staging.

Actual architecture: Orders store flat resolved rate values only (customer_rate_value, vendor_rate_value, driver_rate_value) stamped at order creation from whichever project SRS matches the order's pickup site. There is no order-level SRS management UI or API. The order copy endpoint (POST /orders/{id}/copy) takes {"date": "...", "include_assignee": true} and copies these flat fields + order-level add-on assignments server-side.

  • Phase 3 (11 BLOCKED): All require dispatched jobs — orders were in "created" state only.
  • Phase 4 (6 BLOCKED): 3 run, 3 require completed orders/invoices.
  • Phase 5 (10 BLOCKED): Require completed jobs for invoices/settlements.
  • Phase 6 (5 BLOCKED): Require sub-vendor/driver account login.
  • Phase 7 (18 BLOCKED): Spec expected order-scoped SRS; actual implementation uses flat rates only.
