TC5 Key Finding — Two Distinct Confirmation Dialogs

The star button surfaces two different dialogs depending on direction: "Apply this rate card to all jobs?" (set primary, yellow "Apply to jobs" CTA) vs "Remove default rate card?" (unset, red "Remove default" CTA with warning about job rematching). The star button only renders in the Create/Edit Order form context — it is absent from the project-edit dialog (which shows copy-only buttons).

Key Finding — Rate Matching is Manual

The Rates Experience does not auto-select the "best-matching" rate card at any point in the flow — not in the Create Order modal, not in the Edit Rates dialog. The condition tags (equipment type, material) on rate cards are informational labels that help dispatchers identify the right card; the dispatcher must manually select it. The "most specific match wins" logic from the Rulebook video appears to be a documentation/design intent, not a currently-implemented automatic behaviour in staging.

Bug Surfaced
⚠️ MUI Select open-state leak on Submit Low · UI only
What happens: When a type dropdown is opened and not properly dismissed before submission, the MUI Select retains aria-expanded="true" in React state — even if the listbox is visually gone. The form's Submit button then silently does nothing: no spinner, no error toast, no validation message.
Root cause: The MUI Select open state (a useState(false) hook inside the component) stays true if the dropdown closes via a non-standard path (e.g. the portal container is removed from the DOM before the close handler fires). React's form onSubmit handler likely guards against submitting while a Select is open, but shows no feedback.

Impact on real users: Minimal — a real user clicking a dropdown and then clicking elsewhere will always trigger the proper close handler. This would only surface if a user clicks a dropdown trigger and then immediately hits Submit without selecting an option or clicking away first.

Suggested fix: Add a visible validation state ("Please select a rate type") that appears on submit even when the Select is in an open state, so users aren't left with an unresponsive Submit button.

Defects Found
🐛 Conditions missing from "Freight Rates created/updated" timeline events High
Affects: R-P5.16 · R-P5.24 · TRE-14483 / TRE-14493

When a Site Rate Schedule with Pickup Site, Dropoff Site, Equipment Type, or Material conditions is created (or updated), the project timeline event body renders only Name, Customer rate, Vendor rate, and Driver rate. Condition names are not displayed.
Observed: "Sand + Dump Truck (Specific)" — conditions Standard Dump Truck (Equipment Type) + Concrete Sand (Material) — create event body contains no mention of those conditions.

Expected: Event body should include condition labels, e.g. Equipment Type: Standard Dump Truck · Material: Concrete Sand.

Likely cause: The timeline event template does not include the conditions array from the SRS payload. May be a backend omission (conditions not serialised into the audit log entry) or a frontend omission (data present but not rendered).
🐛 Adding/removing a rate add-on does not log a timeline event Medium
Affects: R-P5.18

When an add-on is attached to (or removed from) a rate's Customer, Vendor, or Driver row and the project is saved, the project timeline receives no new event. The save succeeds (add-on persists), but auditability of add-on mutations is absent.
Observed: Added add-on type addon1-test to "Sand Material Rate" Customer Rate → Save → timeline collapsed + re-expanded → previous event ("Freight Rates deleted" at 2:09 PM) remained the most recent; no "Freight Rates updated" entry appeared.

Expected: A Freight Rates updated event should appear with the add-on change captured (e.g. "Customer add-on: addon1-test added").

Note: The Freight Rates updated event does fire correctly when a rate amount changes (R-P5.17 passes). The gap is specific to add-on mutations — they likely update the rate's add_ons sub-resource without triggering the project timeline writer.
Methodology

Rates were created programmatically via browser automation on app.horizon-staging.com using the christophe@tread.io account. A new project ("cristophe test", PROJ-12615) was created fresh for this test. 50 rate blocks were added via the project edit modal — each with a unique rule name, customer rate ($50–$295 in $5 increments), vendor rate (≈75% of customer), and a cycling rate type across Per Hour, Per Load, Per Ton, Per Tonne, Per Yard, Per Bushel, Flat Rate, and Per Lbs. All 50 were submitted in a single form payload. Network requests and console output were captured via Chrome DevTools Protocol.
