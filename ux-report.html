Section 01

Global UI Issues

Problems that affect every screen — fixing these will lift the entire product immediately, before any screen-specific work begins.

Critical Everything is text. There is almost no visual language.

Across every screen, data is presented as raw inline text separated by bullet characters (•). Status states, data types, amounts, and actions are visually indistinguishable from one another. A dispatcher scanning 10 orders in 30 seconds cannot quickly identify what needs attention — every row demands the same reading effort.

  • Order status is shown as plain text: "Pending" — no colour, no icon
  • Rates are inline: "Customer Rate: $15.00 Per Ton •" — same weight as the project name
  • Truck / load counts use raw numbers with no visual context
  • All action buttons are unlabelled icon buttons with no tooltips visible on scan

Fix: Introduce a consistent colour-coded status system (see §13), replace bullet-separated inline metadata with labelled chips/badges, and use typographic hierarchy (size + weight contrast) to separate primary from secondary information.

Critical No visual hierarchy — every data point has equal visual weight

The most important piece of information on an order card (its status and how many trucks have accepted) competes visually with the least important (the internal requestor name). Everything is rendered in the same font size and colour.

  • Project name, customer, rate, order ID, tags, start time, requestor, truck type, foreman, and pickup site are all presented at the same scale
  • Truly critical fields — order status, truck acceptance rate, and load count — are buried at the bottom of each card

Fix: Apply an F-pattern layout to cards: large primary headline (order name + status), secondary row (customer · project · rate), supporting row (trucks · loads · time), and actions pinned right. See the card mockup in §04.

High Dark palette is correct but not executed — it looks like a plain white app that was just dark-mode-toggled

The dark background is a good foundation for a field-operations tool (reduces eye strain outdoors and on mounted tablets). However the current implementation adds no depth, no surface differentiation, and no ambient lighting effects — it just swaps white backgrounds for dark ones.

  • Cards, sidebar, and main content area are nearly identical in tone — nothing groups or separates them
  • No gradient, shadow, or surface layering to create depth
  • Yellow accent (#FFE500) is used inconsistently — applied to the logo but rarely used to guide attention

Fix: Establish a 3-level surface system (bg → surface → surface-raised), use yellow accents exclusively to highlight actionable primary CTAs, and add subtle card shadows to lift interactive elements off the canvas.

High Icon-only action buttons have no labels — discoverability is broken

On the Dispatch screen and Projects list, multiple icon buttons appear side-by-side with no label and no tooltip on hover. New users have no way of knowing what they do without clicking. On mobile, there is no hover state at all.

  • Order cards have 2 unlabelled icon buttons of unknown purpose
  • Project rows have 3 unlabelled icon buttons
  • "Send All Jobs" and "Export Jobs" are correctly labelled — those buttons demonstrate the right pattern; apply it consistently

Fix: Add visible text labels to all action buttons on list rows. If space is tight, use an overflow "···" menu to consolidate secondary actions.

Medium Inconsistent date formatting across screens

Dates are formatted differently on every screen: "Fri May-22-2026" in the dispatch header, "May-22-2026" on project cards, "May 22, 26" in the approvals grid, "Apr 18, 26" elsewhere. The year is abbreviated in some places and full in others.

Fix: Standardise on a single format. Recommended: 22 May 2026 (full) and 22 May (compact, same year). Never abbreviate the year to 2 digits in a business app.

Medium No loading states, skeleton screens, or transitions

Navigation between sections causes a hard, blank flash while data loads. The Reports screen shows a raw "Loading..." text from the embedded Omni iframe. There are no skeleton placeholder layouts during data fetches.

Fix: Add skeleton card components that mirror the final layout — animated grey blocks that give users a sense of structure before data arrives. This is especially important on the Dispatch and Approvals screens where grids are large.

🎨
Current colour usage: The yellow accent (#FFE500) is used in the logo and a handful of status buttons, but it competes for attention with plain white text. Every accent should earn its place. Yellow should mean "primary action" everywhere — if something yellow isn't clickable or urgent, it's noise.

Section 02

Navigation & Information Architecture

The top tab bar works at a high level but has serious gaps in depth navigation, role-based context, and mobile usability.

High 8 top-level tabs with no icons — scannability collapses past 5 items

The main navigation shows 8 text tabs: Projects · Dispatch · Assigned · Approvals · Live Map · Reports · Sites · Settings. Research (Nielsen Norman) shows that text-only navigation above 5–6 items forces users to read rather than recognise, slowing navigation significantly.

  • No icons to aid quick visual recognition
  • "Assigned" and "Dispatch" are conceptually adjacent — their distinction is unclear without reading documentation
  • No active state beyond a subtle underline — hard to see at a glance where you are
  • The user's company name and role text is buried in the header, adding visual clutter with no actionable purpose
Current — Text-only tabs

Medium No breadcrumb trail inside deep screens

Navigating to a project detail or order detail gives no visual indication of where you are in the hierarchy. There is no back button, no page title, and no breadcrumb. Users must use the browser back button or manually click to the top-level tab.

Section 03

The Missing Dashboard

There is no home screen. Users land directly in Dispatch every session. A contextual dashboard would dramatically improve situational awareness and reduce time-to-action for every role.

Critical No operational home screen — users have no "what needs my attention right now" view

A construction operations platform handles many simultaneous threads: orders needing assignment, shifts needing approval, drivers not yet accepted, billing items overdue. Currently there is no single screen where any of this surfaces automatically — users must navigate to each section manually and look for problems.

  • Dispatchers must go to Dispatch → Orders to see unassigned orders
  • Billers must go to Approvals → Payables/Receivables to see pending items
  • Admins have no KPI summary anywhere
  • No notification inbox or alert centre

Fix: Add a Dashboard as the default landing screen with role-aware content. See the mockup below.

Section 04

Dispatch — Orders View

The most-used screen in the product. Every dispatcher spends the majority of their session here. It is overloaded with undifferentiated text and critical information is hard to find fast.

Critical Order cards are a wall of inline text with no visual structure

Each order card renders all its metadata as a flat horizontal text string: Customer: Account CUSTOMER-123 • Customer Rate: $15.00 Per Ton • Project: PROJECT-A • ORD-26876 • Staging Hauler Co • Prevailing Wage • Pending • Start: 8:00 AM Fri May-22-2026 • Requestor: Christophe Staging • Truck Type: Standard • Foreman: Christophe Staging • Pickup: Koffi • TRUCKS: 1 Truck • 0/1 Accepted • LOADS: 1 Load • 0 • MATERIAL ORDERED: [No Material Qty]

A dispatcher scanning this during a busy morning cannot in under 2 seconds answer the key question: "Which orders are at risk right now?"

  • Status ("Pending") is tiny and unstyled
  • Truck acceptance ratio (0/1) is buried at the bottom
  • Rate info ($15/ton) is given equal prominence as the customer name

High Department filter tabs overflow to a hidden "More" menu — tabs are not the right pattern here

There are 6+ department tabs visible plus a "More" overflow button. This pattern breaks when the number of departments grows. Users with 10+ departments will rarely see all their options.

Fix: Replace the overflow tabs with a multi-select filter dropdown labelled "Departments". Show selected departments as dismissible chips above the order list. This scales to any number of departments and allows filtering by multiple at once.

High Status summary panel at the top is numbers-only — no visual context

The sidebar panel shows: Accepted: 2/11 · Available: 168/170 · Unassigned: 9 · Accepted: 0 · Assigned: 0 · En Route: 0 · Sent: 0 .... These are critical operational numbers but they're rendered as plain text label/value pairs with no visual differentiation.

Fix: Use mini progress arcs and colour-coded counters — red for things requiring action (Unassigned: 9), green for things going well (Available: 168), grey for zero-count items which can be collapsed.

Medium No Kanban / pipeline view option for orders

Orders have 8+ status states (Unassigned, Sent, Accepted, En Route, Loaded, Unloaded, Completed, Cancelled). Currently these are filtered via a dropdown but there is no visual pipeline view showing the flow of orders through these states.

Fix: Add an optional Kanban view toggle — columns for each status state, cards draggable between columns. This is the most intuitive way to understand and manage the order lifecycle. The current list view remains as the default for users who prefer it.

Section 05

Dispatch — Driver Sidebar

The left sidebar driver panel is the primary tool for assigning trucks to orders. Its current design makes it slow to use under time pressure.

High Driver list is text-only — no status, location, or availability signal

Each driver entry shows: truck number · driver name · truck type · company. That's it. There is no real-time status indicator showing whether the driver is online, driving, at site, idle, or unavailable. A dispatcher assigning a driver to an urgent order cannot see at a glance which drivers are free.

  • No colour or icon for driver status (available vs en route vs idle)
  • Long concatenated names like "52.574538 • AB one Player one Standard New Label superexpaliciouscalifragous totallylong new name testing" overflow and wrap unpredictably
  • Checkbox + drag handle is the only interactive element — no quick "assign to order" flow

Fix: Add a coloured status dot to each driver row (green = available, yellow = in transit, grey = offline). Show current job if assigned. Cap name display at 30 chars with an ellipsis.

Section 06

Order Creation — Wizard Opportunity

Creating a new order is one of the highest-frequency actions in the system. The current experience is a long, scrolling form. A guided wizard would reduce errors and training time dramatically.

High New Order is a dense form — no guidance, no intelligent defaults

The "New Order" button opens what appears to be a full-screen form with many fields. Dispatchers creating their 5th order of the morning want speed; new dispatchers need guidance. Neither gets what they need from a flat form.

  • No indication of which fields are required vs optional
  • No smart defaults based on recent orders or project settings
  • No inline validation until submit
  • "Quick Order" and "New Order" coexist — the distinction is unclear

Fix: Replace with a 4-step wizard (or use Quick Order as the primary flow and promote it). Smart defaults from the last similar order. Required fields bold. Optional fields collapsed until needed.

Section 07

Projects List

The projects list is a functional but visually undifferentiated scroll of text rows. With dozens of active projects it becomes a scanning challenge.

High Project cards have no at-a-glance health indicators

Each project card shows name, customer, date, company, and an "Active" status badge. There is no indication of how many orders are running, what the total load volume is, whether there are any issues, or when the last dispatch was.

  • A project manager cannot tell if a project is "active" in name only or actively dispatching loads
  • No visual differentiation between a project with 50 open orders vs one with 0
  • The three unlabelled action buttons (seen on every row) have unknown purpose

Fix: Add a compact metric row to each project card: open orders count, trucks assigned today, loads completed this week. Use a coloured left border to indicate health (green = on track, orange = has unassigned orders, red = has blocked orders).

Medium Status filter tabs (Active / Completed / Inactive / Cancelled) are text-only counts

The four status filter tabs show counts but not proportions. If 90% of projects are Active, the tab label gives no sense of scale.

Fix: Show the count in a coloured badge on each tab. Add a visual bar below the tabs showing the proportion of projects in each state (like a mini stacked bar).

Medium No card vs. table view toggle

Some users (admins reviewing 50+ projects) need a dense table view. Others (dispatchers looking for a specific project) prefer the richer card format. Currently there is only one view with no toggle.

Fix: Add a view toggle (⊞ Cards / ☰ Table) in the top-right of the list. The table view shows columns: Name, Customer, Status, Orders, Open Loads, Last Activity, Actions.

Section 08

Project Detail

The project detail screen (accessed via the project URL) is where most of the operational complexity lives. It needs to surface context, not just data.

High Project detail has no summary header — you land in a data dump

Navigating to a project detail drops users directly into a list of orders with no project-level context visible. The project name, customer, total scope, and current status are not visible without scrolling.

Fix: Add a sticky project header card that stays visible as you scroll through orders: project name + status, customer, dates, estimated setup time, and a 4-metric summary bar (total orders, trucks assigned, loads completed, revenue to date).

Medium Timeline / activity feed is a modal — it should be a sidebar panel

The project timeline (showing SRS changes, order updates, etc.) is accessed via a "Show timeline" button that opens a separate view. This means leaving the order list context every time you want to check history.

Fix: Add a collapsible timeline panel on the right side of the project detail. It can be toggled open/closed without leaving the order list. Show the 10 most recent events, with "Show all" expanding to the full history.

Section 09

Approvals

The Approvals screen is the highest-density grid in the product — 14 columns of driver shift data with action buttons in the last column. It is functional but visually exhausting.

High 14-column data grid is unreadable at a glance — critical columns have no visual priority

The Approve Work grid shows: Date, Driver, Truck No., Vendor, Shift Start, Shift End, Break, Total Hrs, Job Ct, Load Ct, Tickets, Materials, Shift Status, Actions — 14 columns side by side with equal visual weight. A biller approving shifts must read every column every time.

  • Shifts with anomalies (e.g. 16-hour shift that is almost certainly a data error) look identical to normal shifts
  • Missing values rendered as plain "-" — indistinguishable from intentional dashes
  • Shift Status column contains "End Shift" button (an action) and "Approved"/"Approve Shift" (mixed action + status) — conceptually inconsistent
  • No colour highlighting for anomalous values (Total Hrs: 16.00 should be flagged)

Fix: Highlight anomalous values in orange/red (shifts over 14 hours, missing truck numbers). Add a status column with coloured pills. Move "End Shift" out of the grid into the row's overflow menu.

Medium Bulk approval is not visible — users must approve shifts one at a time

There are no checkboxes on the grid rows for bulk selection. A biller with 20 straightforward shifts to approve must click "Approve Shift" 20 times. There is no "Select All" or "Approve Selected" action.

Fix: Add row checkboxes and a floating bulk action bar that appears when rows are selected: "Approve Selected (12)" and "Export Selected". This is the standard pattern used by Gusto, Rippling, and other payroll-adjacent tools.

Medium Payables and Receivables tabs are text labels with no counts

The Approvals section has four tabs: Approve Work, Approve Jobs, Payables, Receivables. None show a count badge indicating how many items need attention. A biller arriving at this screen has no way to quickly prioritise which tab to check first.

Fix: Add count badges (red if > 0) to each tab: Approve Work (5), Payables (12), Receivables (3). This pattern is universally understood and immediately communicates urgency.

Low Expand button reveals detail panel but its content is unknown until clicked

Each row has an "Expand" button that opens a detail panel. Users cannot tell what additional information will appear without clicking it — there is no preview or tooltip.

Fix: Add a tooltip on hover: "Show load details" or "Show job breakdown". Alternatively, show a subtle preview of the expanded row's key data (e.g., job names) as a second line in the collapsed row.

Section 10

Settings

Settings is a flat scroll of 22 items in the left nav, each pointing to a simple form. It gets the job done but feels like a legacy admin panel rather than a modern product.

High User profile is a read-only key-value list — a table masquerading as a form

The User Profile screen shows Basic Information as a series of heading/value pairs: First Name: Christophe, Last Name: Staging, Email: christophe+staging@tread.io, Phone: Not specified, Role: Company Admin..., etc. Below it are 14 notification settings all formatted identically as heading/value pairs.

  • No visual separation between field groups
  • Notification settings (Push, None selected) are plain text with no toggle switches or checkboxes visible without clicking Edit
  • The "Edit" + "Update" button pair at the top is easy to miss

Fix: Use an inline edit pattern — clicking a field value turns it into an input. Notification settings should use visible toggle switches (on/off) rather than text that says "Push" or "None selected". Group notifications into logical categories (Jobs, Orders, Compliance).

Medium 22-item flat settings nav has no grouping — finding anything requires reading every item

The left navigation lists: General Settings · User Profile · Users · Drivers · Customers · Vendors · Equipment Types · Equipment · Materials · Rates · Sites · Departments · Labels · Overtime Rules · PW Benefits · Service Classes · Tax Rate Schedules · Document Types · Configuration · Integrations · Agents · API Clients. No headings. No grouping. No icons.

Fix: Group with section headings and icons. Add a search box at the top. Proposed groups:

  • 👤 Account — General, Profile, Users
  • 🚛 Operations — Drivers, Equipment Types, Equipment, Materials, Sites, Departments, Labels
  • 💰 Finance — Rates, Overtime Rules, PW Benefits, Tax Rate Schedules
  • 🔗 Integrations — Integrations, Agents, API Clients
  • ⚙ Advanced — Configuration, Service Classes, Document Types

Section 11

Reports

Reports uses an embedded Omni Analytics iframe — the analytics engine itself is good but the integration into the product feels bolted-on rather than native.

Medium Embedded iframe feels disconnected from the rest of the product

The Reports page shows a left sidebar with 5 navigation buttons (Dashboard, Projects & Orders, Dispatch, Billing, API Integrations) and then immediately fills the rest of the page with an embedded Omni iframe. The Omni frame has its own loading spinner, its own colour scheme, and its own "Powered by Omni Analytics" branding — creating a jarring context switch.

  • Users feel like they left the product when they enter Reports
  • The Omni loading state ("Loading...") clashes with the surrounding Horizon UI
  • No print / export buttons visible at the Horizon level — buried inside the Omni frame

Fix: Wrap the iframe in a styled container that blends with the Horizon theme. Add Horizon-branded export/share buttons above the iframe that proxy to the Omni actions. Use the Omni embed styling API to set fonts and colours to match Tread's palette.

Low Report navigation (5 items) uses full-width buttons — wastes vertical space

The 5 report category buttons (Dashboard, Projects & Orders, Dispatch, Billing, API Integrations) are full-width and tall. On a 1440px wide screen this means the first 60px of vertical space is consumed by navigation before any data appears.

Fix: Move report navigation to a tab strip at the top of the reports area, matching the tab pattern used elsewhere in the app. Free the vertical space for the data.

Section 12

Empty States & Error Pages

High 404 / not found page is a plain "🚧 We couldn't find that page" message

The Sites tab and the /admin/companies route both return the same 404 screen: a construction cone emoji, a heading, and a subtitle. There are no suggested links, no search box, and no navigation back to a valid page. Users who land here are completely stuck.

  • No "Back to Dashboard" button
  • No "Search for what you're looking for"
  • No explanation of why access was denied (permission issue vs. wrong URL)

Fix: Design a proper 404/403 page with: the user's avatar and company name (confirm they're logged in), a large friendly message, 3–4 suggested quick links (Dashboard, Dispatch, Projects), and a "Contact support" link.

Medium No empty state for orders list when filters return no results

If department filters or date filters produce no results, the list area silently empties. Users cannot tell if there is genuinely no data or if they have an active filter narrowing results too far.

Fix: Show an empty state illustration with context: "No orders found for this department on 22 May. Clear filters or create a new order."

Section 13

Recommended Visual Patterns

A consistent component vocabulary would solve most of the text-density problems across the entire product. These are the five highest-value components to build first.

Pattern A Status Pill System — replace all plain-text status labels

A single, consistent status pill with a colour dot, label, and subtle background. Used on orders, projects, drivers, shifts, and loads. The colour encodes urgency instantly without reading the word.

Pattern B Metric Mini-Card — replace inline counts with scannable cards

A compact card showing a number, label, and optional trend indicator. Used in the dispatch sidebar, project cards, and the dashboard header.

Pattern C Inline Field Chips — replace bullet-separated metadata

Instead of "Customer: Account CUSTOMER-123 • Customer Rate: $15.00 Per Ton • Project: PROJECT-A", render each piece of metadata as a small labelled chip. Visually scannable, touch-friendly, consistent across all cards.

Pattern D Notification Badge — count overlays on nav tabs and action buttons

Red count badges on nav tabs that have pending actions. Approvals tab shows "5" if there are 5 shifts awaiting review. Dispatch tab shows the unassigned order count. This follows every modern SaaS app and eliminates the need to visit each section to discover if action is needed.

Pattern E Contextual Action Bar — floating bar for bulk and primary actions

A bar that appears at the bottom of the screen when items are selected (approvals, orders, projects). Shows "X items selected" and offers relevant bulk actions. Disappears when selection is cleared. Used by Linear, Notion, GitHub, and virtually all modern data-heavy applications.
